# What Nobody Tells You About Running Bedrock in Production

The insight leg of my platform calls a foundation model on Amazon Bedrock to explain unusual market moves. It has been in production for weeks. In all that time, not one insight has come from the model. Every single one is deterministic, generated by a rule-based fallback, and the system is healthier for it. This is the story of why, and of the specific way Bedrock can be approved, authorized, entitled, and still refuse to run.

## The setup, so the failure makes sense

The insight Lambda calls `InvokeModel` with the Anthropic messages format. It does not ask the model to "please reply in JSON". It forces a single tool through `tool_choice`, whose input schema pins the reply to exactly four fields: a signal enum of BULLISH, BEARISH, or NEUTRAL, a confidence in the zero-to-one range, a rationale string, and a list of drivers, all required. The reply is read from the tool-use content block and validated by a record type; invalid output is retried once and then falls back. The model supplies only judgment. The service stamps the ticker, timestamp, model id, and prompt version itself, so the model never invents identity or time.

I mention the tool-use design first because it is the part everyone writes tutorials about, and it is the part that gave me the least trouble. Structured output through a forced tool is genuinely reliable. Getting the model to run at all is where production diverges from the tutorial.

## The saga: approved, authorized, and still denied

Here is the sequence, because the sequence is the lesson.

First I had the wrong model id. The insight service pointed at `global.anthropic.claude-sonnet-4-5-20250929-v1:0`, a cross-region inference profile id that, it turned out, never existed in ap-south-1 at all. What masked the mistake for a while is instructive: an availability check against that non-existent id still returned region and authorization data without erroring, so the id looked valid when it was not. Running `aws bedrock list-inference-profiles` showed no such entry. I corrected the environment variable to `global.anthropic.claude-sonnet-4-6`, which does exist and is active in the region, and the Lambda's IAM grants both the profile and the wildcard-region foundation-model ARNs.

Fixing the id did not fix the leg. Every invoke against the correct id came back with `AccessDeniedException: INVALID_PAYMENT_INSTRUMENT`.

This is the part nobody tells you. The console showed model access as granted. The region showed the model as available. The IAM role had the invoke permission on the right ARNs. By every signal a tutorial teaches you to check, the setup was correct. And the account still could not invoke, because the AWS Marketplace subscription for Anthropic models needs a valid payment method on file in the billing console before Marketplace will even evaluate the agreement or the quota. Model access approval and a working payment instrument are two separate gates, and the second one throws an error worded like an authorization failure. I chased the wrong layer for a while because the error says "access denied", not "add a card".

I had also, earlier, read the symptom two other ways: once as an incomplete Marketplace agreement, once as a zero-tokens-per-day quota. Both readings pointed at real facts, but both were downstream of the same root cause. There was no payment method, so Marketplace never got as far as the agreement or the quota. Three plausible diagnoses, one underlying gap. That is worth internalising: Bedrock's failure surface does not cleanly map one error string to one cause, and you can spend real time fixing symptoms that were never the disease.

I looked for a substitute in the same account. First-party Nova models are entitled there, no Marketplace subscription required, so for a moment they looked like the escape hatch. They are daily-token-throttled to the point of being unusable for a real workload, throwing throttling exceptions almost immediately. Entitled but throttled is not a working substitute for approved but unpayable. So the primary path stayed dark, and it is still dark as I write this: the payment fix is account-side and outside my code.

## Cross-region inference profiles, briefly

One thing that did work as advertised is the cross-region inference profile itself. A caller in ap-south-1 cannot always invoke a model that is not locally on-demand invocable; the `global.` profile id routes the request to whichever region can serve it. That indirection is the correct way to reach Claude from a region that does not host it directly, and it is why the model id carries a `global.` prefix rather than a bare model name. The lesson from the id confusion is narrower than "profiles are hard": it is that you must confirm the exact profile id exists in your region with `list-inference-profiles`, because an availability check will lie to you about an id that was never real.

## The fallback that carried production

Here is the decision that actually mattered. I designed the LLM path so the model is an enhancement, not a dependency. When Bedrock cannot be used, a rule-based generator produces an insight in the exact same four-field schema, and it fires on any of the failure modes: the cost breaker being open, a throttling exception, a model timeout, invalid output on both attempts, or any other Bedrock error at all.

The rule-based logic is deliberately simple and deterministic. Price above the 52-week high signals bullish, below the low signals bearish, otherwise a change of at least one percent in either direction sets the direction and everything else is neutral. Confidence is fixed at 0.4, deliberately lower than an LLM insight would carry, so a consumer can tell the two apart at a glance. The drivers are built from the anomaly reason, the change percent, the volume, and any 52-week break. The rationale is a template that names the ticker, the signal, and the move, and states plainly that it came from static thresholds because the model was unavailable. No randomness, so the same input always yields the same insight.

The honesty of the design is a single attribute on the stored item: `source`, either `BEDROCK` or `RULE_BASED`. Nothing lies about which path produced an insight. The frontend even drops the model id from the footer when the source is rule-based, because there is no model to credit.

What this bought me is the thing I want to convince you of. A real, weeks-long outage of the primary path produced exactly zero dead-letter entries. Not one insight failed into the DLQ over the Bedrock denial. The platform absorbed a total failure of its headline feature without a single user-visible error, because the headline feature was never load-bearing. The moment the account's payment method is fixed and access re-requested, the switchover is automatic: the service tries Bedrock first, and only reaches for the fallback when the try fails. No deploy, no flag flip, no code change. The fallback is not a degraded mode you toggle into; it is the default safety net that the model sits on top of.

## Cost control, because an LLM path that can run must be capped

The other thing nobody tells you is that the moment the model does start working, it becomes the most expensive line in the system, and you want the cost controls already in place before that happens rather than after the first surprising bill.

I built three layers, from cheapest to last resort. The anomaly gate is first: the pipeline only calls the insight leg when a price move is statistically unusual, so most ticks never reach the model at all. A per-day circuit breaker is second: the cost tracker writes a per-invocation record with input and output token counts and a computed cost, and increments a daily counter under a date-keyed partition. The breaker opens when the day's spend reaches a five-dollar cap, after which the insight service skips Bedrock entirely and serves the fallback until the UTC date rolls over into a fresh partition that starts at zero. SDK-level retries are disabled through a no-retry policy, because Step Functions already owns retry and letting the SDK retry underneath it would only amplify token cost. Beyond those, AWS Budgets and ephemeral teardown are the outer fences.

Two details in the cost layer are the kind you only get right by thinking about the failure. The breaker records cost before it validates the model output, because tokens are billed whether or not the reply parses, and a cost tracker that only counts successful parses would undercount exactly when the model is misbehaving. And the breaker read is intentionally eventually-consistent and fail-open: if the counter read itself fails, the breaker stays closed and insight generation proceeds, because a DynamoDB hiccup blocking every insight is a worse outcome than a slight cost overshoot under burst. The soft cap may overshoot the five-dollar line marginally under concurrency, and that is the correct trade for a test-window budget.

## The lesson

The thing I would tell anyone putting a model behind a production feature is this: assume the model will be unavailable, and design so that unavailability is boring. Not a page, not a DLQ entry, not a 500, just a lower-confidence answer tagged honestly with its source. My account's payment problem is unusual, but the shape of the failure is not. Models throttle, quotas exhaust, regions have outages, and Marketplace subscriptions lapse. If any of those can take down your feature, the model is a dependency and you have bought yourself an incident waiting for a bad day. If none of them can, because a deterministic path always answers in the same schema and the model is only ever an upgrade over it, then you have bought yourself the freedom to run in production for weeks with the fancy part switched off and nobody the worse for it.

The rule-based fallback is not the unglamorous part of this system that I am embarrassed to admit is doing the work. It is the part that let me ship at all.
