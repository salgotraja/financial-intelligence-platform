# How I Built an Observability Layer for a Serverless System That Actually Helps During Incidents

Most observability write-ups show you a dashboard and stop. A dashboard is where an investigation ends, not where it starts, and the honest test of an observability layer is not how it looks on a resume screenshot but whether it points you at the cause when something breaks. This platform gave me an unusually good testbed for that, because a whole class of its bugs could not be seen at all until real AWS ran the code. Units passed. LocalStack passed. The synth was clean. And then the first real deploy, or the first real CI run, surfaced a defect that no amount of local testing could have caught. Each one taught me something about what to instrument and where to look.

## The layer, briefly

Every Lambda has its own log group and every request logs a correlation id you can grep across services. There is one CloudWatch dashboard aggregating API latency at the 50th, 90th, and 99th percentiles, request and 4XX and 5XX counts, Lambda errors and throttles and duration and concurrency, ingestion executions, DLQ depth and age, and DynamoDB capacity and throttles, ending in an alarm-status widget.

The alarms follow one rule: page on symptoms, not on causes. Two P1 alarms publish to a critical topic: API p99 latency above 500ms for five minutes, and 5XX rate above one percent. Two P2 alarms ticket to a warnings topic: Step Functions execution failures, and any DLQ depth above zero. What is deliberately not an alarm is as important as what is. 4XX counts, per-Lambda throttles, and DynamoDB throttles all live on the dashboard as diagnostic context but never page, because they fluctuate naturally and paging on them trains you to ignore the pager. The 5XX alarm carries a metric-math floor, firing only when the error count itself clears five, so a single stray error on an idle system never wakes anyone. Every alarm carries a structured description naming the symptom, the likely causes, and the first action, so an alert is a starting point rather than just a noise.

On tracing I made a specific call, recorded in ADR 0001 and shipped in commit a03a38f: the platform uses only the managed trace path. `Tracing.ACTIVE` in CDK makes the Lambda platform, API Gateway, and Step Functions emit trace segments themselves, so there is no deprecated X-Ray SDK in any JAR, and the root pom deliberately manages no such library. When code-level custom spans become necessary, the agreed direction is OpenTelemetry exporting to CloudWatch, not a self-managed backend whose always-on compute does not fit this account's budget. The point of an observability decision is not to install every agent available; it is to instrument at the platform layer where you get traces for free and carry no deprecated dependency.

## The class of bug that only real AWS can show you

Here is the spine of the whole discipline. Some defects are structurally invisible to unit tests and to LocalStack, and only the real service, or a real off-hours CI run, exercises the behaviour that breaks. Every one of these cost real debugging, and every one was caught by a specific alarm, smoke gate, or browser check rather than by a test.

The first I ever hit was a Cognito custom-attribute name over 20 characters. Synth passed; the deploy failed validating the user pool, because that limit is enforced only against the real API at deploy time. `cdk synth` passing is not proof a resource is valid.

Then came a caching defect that a smoke test caught. With the stage cache on, every ticker returned the same insight, whichever was cached first, because a path parameter is not part of the cache key unless you declare it. `/insights/RELIANCE.NS` and `/insights/TCS.NS` had collapsed to one cache entry. The fix was to declare the ticker as an explicit cache key, and the lesson generalised: a response cache is only correct when the key includes everything that varies the response.

The third was an authorizer that API Gateway rejected because a null field, an IAM policy `Condition`, serialised into the policy document when it should have been omitted. The SDK event POJOs carry no Jackson inclusion annotations, and the function's mapper includes nulls by default, so the null slipped through and API Gateway refused the malformed policy. I stopped trusting the SDK object to serialise cleanly and hand-built the policy response as a map, so no null can slip in. When the wire format must be exact, control it explicitly.

The fourth is the SnapStart trap I alluded to in the architecture write-up. A Spring Boot Lambda was cold-starting in 5 to 10 seconds despite SnapStart being configured, because SnapStart snapshots a published version and I was invoking `$LATEST`, which gets no snapshot and runs the full Spring init every time. Publishing a version and pointing an alias at it fixed it. The SnapStart win is real but only through the alias indirection.

The fifth is my favourite, because the log lied to me. An authenticated read returned HTTP 500 with a mapped error body, yet the query Lambda's own log showed a clean run: insight found, request complete. Everything around the integration looked healthy. The cause lives only inside real API Gateway: for a non-proxy Lambda integration, the gateway picks the integration response by matching every selection pattern against the invocation's error message, and on a successful invocation that error message is the empty string. My catch-all 500 pattern ended in `)*$`, and the star quantifier matches the empty string, so the 500 response outranked the patternless 200 default on every single success. The fix was one character, `)*$` to `)+$`, requiring a non-empty error message. LocalStack does not evaluate that regex-versus-empty rule, so the defect was invisible until the first real deploy. The rule I took away is a debugging one: when a 500 appears with a mapped error body but the Lambda log is clean, read the integration response config, not the handler.

There were more of this family. Re-creating a Cognito pool invalidates every cached token because the pool is not restored but rebuilt with new ids and empty user lists, so every authenticated call returns Unauthorized after a Security redeploy even though the deploy succeeded. A percent-encoded path parameter, `%5ENSEI` for `^NSEI`, arrived at the Lambda still encoded and got rejected by an allowlist that already permitted the caret, because I was validating a value I had not decoded yet. And a CORS defect that only a browser could reveal: CDK's preflight option wires only the OPTIONS response, so the non-proxy method responses never emitted the allow-origin header and every browser fetch failed while curl and the smoke script stayed green. I found that one by driving the actual app in a headless browser, fixed it in commit 8774078 with origin-aware response helpers, and pinned it with a test asserting every response carries the header. A follow-up in 9bdfd54 covered the gateway responses, the 401 and 403 and default 4XX and 5XX, which are a separate surface the method-response CORS never touches.

The through-line across all of these is that the observability layer earned its keep by making each defect fast to locate once it manifested. The 5XX alarm and the smoke gate said "the read path is broken"; the clean Lambda log plus the mapped 500 body said "look at the integration, not the code". The browser QA step, not any test, was the only thing that could surface the CORS gap. Coverage of the real surface, plus the right vantage point when something fires, is what an observability layer is for.

## The two the closure pass surfaced

Two more defects landed during the final validation pass on 2026-07-15, and both are the same shape as the rest: invisible until a process outside my laptop ran the code.

The first defeated drift detection itself. A `cdk diff --all` taken immediately after a clean, successful deploy was not empty. Two consecutive synths of identical source produced order-only permutations across four stacks: the erasure state machine's task parameters, the dashboard widget metric lists, the Lambda environment blocks, request parameters, a stream filter. The cause is that `Map.of` deliberately randomises iteration order per JVM run through a salt seeded at class init, so callers never rely on order. Each `cdk synth` is a fresh JVM with a fresh salt, and that order gets serialised straight into a CloudFormation string property or into the hash that mints an API deployment id, so the template really was byte-different on every synth, churning the workflow, the dashboard, and a replacement API deployment on every deploy and permanently defeating drift detection. The fix, in commits 8bcae98 and 265725f, was an insertion-ordered map helper applied to every multi-entry map that feeds synthesized content, verified by a double-synth producing byte-identical templates for all six stacks and by a post-redeploy `cdk diff --all` finally reporting zero stacks with differences. The lesson is one I will carry to every IaC project: never let hash-map ordering reach anything serialized deterministically downstream, and keep `cdk diff` on a deployed clean tree as a determinism smoke test, because "it works on my machine" hides here. A single process is self-consistent; the nondeterminism only shows across processes, which is exactly what a fresh synth or a CI run is.

The second was a test that read the wall clock. The first real CI run after the billing unlock failed one integration test, and every prior local run had passed. Nothing in the code had changed. The test drove the scheduled-ingest path, which the market-hours guard no-ops outside NSE session hours through an injected clock, and the test never overrode that clock, so it exercised a real, unpinned instant. Local runs had happened to execute during market hours and passed by coincidence; CI ran at 15:38 IST, three minutes after the close, so the guard correctly no-op'd and the "was it stored" assertion failed. The bug was latent in every run; the hour of day decided pass or fail. The fix in commit 3fbb839 pins the clock to a fixed in-session instant and adds a second test pinned after the close that asserts the no-op branch. Any test whose subject reads time, randomness, a locale, or a timezone must inject a fixed value, or it is not a test of behaviour, it is a test of when you ran it. A green suite that has only ever run inside the happy window is hiding a landmine for the first off-hours CI run.

## What k6 and CloudWatch percentiles actually showed

The last piece is load testing, and it taught me to read two different percentiles for two different questions.

A stock run of the load script at 10 rps for 60 seconds over eight tickers reported a 12 percent failure rate, and every one of those failures was a WAF rate-limit block: the script from a single IP now exceeds the 300-requests-per-five-minutes rule, so the WAF working correctly reads as failures in the client. The latency was bimodal, p50 at 40ms and p90 at 83ms, but p95 at 6.3 seconds and p99 at 6.8 seconds, and the tail was not the steady-state cost of the system. It was SnapStart restore ramp: all the function versions had been republished by the deploy about forty minutes earlier, so the run hit fresh versions restoring their snapshots under load. The CloudWatch server-side latency percentiles told the same story from the other side: per-minute p99 ramped through buckets from about 7 to 11 seconds early, then settled to a steady-state floor of 15.5ms. I confirmed SnapStart was engaged by finding the restore records in the logs, restore duration around a second, on a live alias pointing at a published version with optimisation on.

The clean number came from a WAF-compatible rerun at 1 rps warm: 61 of 61 requests fine, zero failures, client-side p99 of 310ms including client round-trip, comfortably inside the 500ms SLO. An earlier ephemeral load test on the read path had shown the same steady-state character, 601 requests with zero failures and a server-side p99 of 110ms.

The lesson in the numbers is that a single p99 figure is meaningless without knowing whether you are measuring the ramp or the steady state, and whether your own load generator is tripping a defence you built on purpose. The bimodal tail is not a latency problem to fix; it is SnapStart doing exactly what it does, paying the restore cost once per fresh version and then serving flat. Read the steady-state percentile for the SLO, read the ramp separately so you understand the cold edge, and know that a 10 rps single-IP run against your own WAF rate rule will draw 403s that are the rule working, not the system failing.

## What actually helps during an incident

Pulling it together: the dashboard is the least important artifact here. What helped, every time, was the combination of alarming on user-visible symptoms so the pager means something, keeping the correlation id greppable across services so one request is one thread, instrumenting at the platform layer so traces cost nothing and carry no deprecated agent, and treating smoke gates and browser QA and the first off-hours CI run as first-class members of the observability layer rather than as tests that happen to run elsewhere. Half the defects in this system were structurally invisible until real AWS or a real CI schedule ran the code. The observability layer's job was never to look impressive. It was to be standing in the right place, watching the right symptom, when one of those finally showed itself.
