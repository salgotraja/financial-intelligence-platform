# Building a Production-Ready Serverless Financial Intelligence Platform on AWS: Architecture Decisions

I built a platform that ingests Indian market data every few minutes, decides whether a price move is interesting, explains the interesting ones, and serves those explanations through a low-latency API with consent, erasure, and audit built in. Everything runs on AWS, entirely serverless, in ap-south-1. This is not a feature tour. It is the set of decisions I made, the reasons behind each one, and the trade-offs I accepted, written from inside the code rather than from a whiteboard.

## Why serverless at all for this workload

The workload is bursty and idle by turns. A scheduled rule wakes the pipeline every five minutes during market hours, fans out over a handful of tickers, and then does nothing until the next tick. The read side is quiet most of the time and spikes when someone opens a dashboard. Paying for idle servers to sit through the gaps between ticks made no sense, so every unit of compute is a Lambda: seven function handlers plus a Cognito trigger, each a Spring Boot fat JAR on the `java25` runtime, invoked through the Spring Cloud Function AWS adapter and selected by the `SPRING_CLOUD_FUNCTION_DEFINITION` environment variable.

The counter-argument to Java on Lambda is cold starts. Spring context initialisation costs 5 to 10 seconds, which is unacceptable on a user-facing read. SnapStart is what makes the choice viable: it runs that initialisation once at publish time, snapshots the initialised memory, and restores every future cold start from the snapshot, cutting the cold start to a few hundred milliseconds. Every function sets `snapStart(ON_PUBLISHED_VERSIONS)`. There is a catch I learned the hard way and will return to in the observability write-up: SnapStart only engages on a published version, so each hot-path function is invoked through a `live` alias, never through `$LATEST`. The alias indirection is load-bearing, not decoration.

## The shape: CQRS as two physical stacks

The single mental model that makes the rest of the layout follow is that this is a CQRS system, and the split is physical rather than conceptual. The write path (a schedule wakes up, fetches data, decides if it is interesting, generates an insight, stores it) is the entire Ingestion stack. The read path (a user calls an API, a Lambda reads the stored insight, returns it) is the Query stack. The two never touch at runtime. They are different CloudFormation stacks with different IAM roles: the write side holds read-write DynamoDB grants and can invoke Bedrock; the read side holds a read-only grant and cannot invoke anything.

I chose this because it keeps user-facing latency predictable regardless of what the write path is doing. A slow model call or an ingestion burst cannot slow a `GET /insights/RELIANCE.NS`, because that request runs none of the write code. It also lets me scale, cache, and grant permissions for each side on its own terms. The read Lambda is read-only by IAM, so a bug there physically cannot corrupt data. The cost of the pattern is that the two sides share one surface, DynamoDB, and I had to model that surface so both sides read it cheaply.

The full deploy topology is six stacks: Data (the only stateful one), Network, Security, Ingestion, Query, and Realtime. Data and Security are `RETAIN`; the rest are ephemeral and torn down between sessions, which matters on a learning account with a five-dollar budget. The NAT gateway is the one thing billing around the clock, which is precisely why the Network stack is disposable.

## The pipeline: EventBridge to Step Functions to Lambda

The write path starts with an EventBridge rule that carries no ticker, only a fixed input `{"source":"eventbridge-schedule"}`. That source tag turns out to be one of the more important design details in the system, and I will come back to it. The rule starts a Standard Step Functions state machine rather than invoking a Lambda directly.

I reached for Step Functions because orchestration logic belongs in the workflow, not buried in Lambda code. The machine reads the watchset (a DynamoDB `Query`, not a Lambda), then fans out over the tickers in a Distributed Map at `maxConcurrency(2)` and `toleratedFailurePercentage(100)`. Each map item runs Fetch, then a Choice on the anomaly verdict, then either Generate Insight or a skip. Retries are a shared policy: three attempts, two-second base, doubling backoff, full jitter. Per-item failures route to an isolated DLQ path so one bad ticker never fails the whole run; a read or map-stage failure fails the run to a separate DLQ.

Two of those numbers are decisions worth defending. The tolerated-failure percentage is 100 because a single unreachable ticker should never lose the good data for every other ticker in the same pass. The concurrency of 2 looks timid until you know the context: this account's Lambda concurrent-executions quota is 10, the minimum tier, shared across every function in every stack. A burst at 5 once starved the API Gateway authorizers into simultaneous failures, so I dropped the map to 2 to keep the pipeline's own burst under half the pool. I chose Standard over Express because Distributed Map is not supported on Express at all, and because I wanted exactly-once semantics and a visible per-state execution history I can open in the console.

The schedule itself encodes a cost decision. Dev runs on a plain rate so development gets a steady trickle. Prod runs a market-hours cron, `MON-FRI` inside the NSE window expressed in UTC, which drops roughly three-quarters of the weekly invocations. Outside trading hours the price is flat, the z-score is always zero, and no anomaly can fire, so running the pipeline then would only burn money.

## The single table

There is one operational table, `financial-platform-dev`, and every entity lives in it: market points, baselines, insights, cost records, the ticker union, watchlists, consent, and correlation groups, all keyed by generic `PK` and `SK` columns with the entity type encoded into the key values. Billing is on-demand, so there is no capacity to plan. There is a single empty `GSI1` reserved for a specific access pattern I will describe shortly.

I picked single-table design because it trades up-front modelling effort for one physical table, predictable single-partition reads, no cross-table joins, and one thing to secure, back up, and monitor. The canonical read is a good example of the payoff: the latest insight for a ticker is `PK = TICKER#{ticker} AND begins_with(SK, 'INSIGHT#')`, sorted descending, limit one. Because ISO-8601 timestamps sort lexicographically, "newest" is just "last sort key", so the hottest query in the system is a one-item read with no index and no scan.

The trade-off is honest: one table with several item shapes is harder to read at a glance, and it demands you lock the access patterns before writing code. TTL does the housekeeping for free, expiring hot market points after 24 hours and insights after 7 days while the baseline never expires. One subtlety bit me and lives in the code as a comment: DynamoDB deletes expired items within a few days of expiry, not at the instant, and an expired-but-undeleted item still comes back from a query, so a newest-first read can serve a formally expired insight unless you screen the ttl attribute.

## Correlation grouping: the insight unit is a group, not a ticker

The most interesting design call on the write side is that an insight is about a group of correlated tickers, not a single one. A third EventBridge rule, `financial-correlations-schedule-dev` on `cron(0/15 3-9 ? * MON-FRI *)`, drives a correlation Lambda every fifteen minutes during the session. It reads recent hot points, converts prices to simple returns so correlation measures co-movement rather than co-level, computes pairwise Pearson correlation for every pair, then runs union-find over every pair whose absolute correlation clears 0.6. Each connected component becomes a group, transitive by construction: if A moves with B and B with C, all three land together even when A and C individually sit just under the threshold.

I chose union-find over a clustering library because grouping is a connected-components problem on a thresholded graph, which union-find solves in near-linear time with no dependency and one tunable. The group id is a SHA-256-derived hex of the sorted member set, so the same membership always yields the same id and passes overwrite idempotently instead of churning ids. The write path then generates one cross-ticker insight per group and mirrors it to each member ticker through `GSI1` (that reserved index, finally earning its keep), so a user's watchlist tickers map to the group insights that touched them, and `GET /insights` dedupes the group insight to one entry per feed.

There is a crash-safety ordering here I settled after a review pass, recorded in commit 13f409b: the `GROUPSET` distinct-index mirror is written before the `GROUP#{id}/META` item, and a group's old insight items are purged on any membership change. If a pass dies mid-write, the authoritative per-ticker reverse lookup still resolves and a stale group insight never outlives the group that produced it.

## The privacy architecture, built in rather than bolted on

Which stocks a user watches is personal data, so the platform treats the Digital Personal Data Protection Act obligations as features. Consent lives in the table as `USER#{sub}/CONSENT`, read strong-consistent on every request, and gates the watchlist fail-closed: an absent record, a false flag, or any read error denies. A signup seeds a default-deny record through a Cognito PostConfirmation trigger.

Consent also gates login through a preAuthentication trigger, and getting that gate right forced a real decision, documented in ADR 0002. The shipped consent record has no status field, so a brand-new user and a user who withdrew both carry `consentGiven=false`. The only signal separating them is whether a `version` was ever stamped. A naive gate that denied every false record would lock a new user out of the very login they need to reach the consent screen, an onboarding deadlock. So the gate allows the never-consented case (version null), denies the withdrawn case (version present), allows a current consent, and denies a stale version after writing a re-consent audit event. The login gate fails open while the data gate fails closed, and the two opposite directions are deliberate: availability where nothing is at risk, enforcement where data is.

Erasure is a Step Functions workflow, not a synchronous handler, because the cascade crosses DynamoDB, S3, Cognito, and SES and I wanted per-step retry, a visible history, and a `deletion_pending` fence against concurrent writes. The ordering is mark-pending, then the reversible DynamoDB deletes, then the irreversible Cognito delete, then the confirmation email, then the audit write last so the record documents a completed erasure. The `StartExecution` name is `sha256(subjectSub#requestedAt)`, injective per request, so a duplicate `DELETE` collapses to an idempotent 202 instead of starting a second erasure.

The audit table carries two record shapes, a decision I reasoned through in ADR 0003. Per-user records keyed by raw `sub` power a subject's own data export. Hashed records keyed by `AUDIT#{type}#{date}`, carrying only a subject hash and actor hash and no raw identifier, are the permanent compliance proof. One partition key cannot be both queryable-by-subject and free of PII, so the honest design writes two records rather than pretending one can serve both readers. Both live in the same append-only table under a `PutItem`-only IAM grant, and both survive the erasure they document, which is the lawful retention exception.

That `PutItem`-only grant is the whole point of the least-privilege posture. The audit table is tamper-evident not because of application code but because no role can do anything except append to it. The query Lambda cannot corrupt data because its role is read-only. The high-privilege Cognito delete lives on the DSR role alone. I enforce these invariants at the IAM boundary, where a future bug in a service method cannot bypass them.

## WAF, and where it does not reach

The REST API sits behind a regional WAFv2 Web ACL: the AWS common and known-bad-inputs managed rule groups, plus a rule that blocks any source IP exceeding 300 requests in a rolling five-minute window. The default action is allow. One honest limitation: WAFv2 has no association target for an API Gateway WebSocket API, so the realtime socket cannot carry a Web ACL. The connect path is still authorized by a Cognito JWT and subscriptions are bounded by a two-hour TTL, but there is no WAF layer for it, and I decided not to pretend otherwise.

## What the shape bought me

The decisions compound. CQRS as two stacks gave me independent scaling and an IAM-enforced guarantee that reads cannot write. Single-table design made the erasure cascade a single prefix query. Keying all personal data under `USER#{sub}` is what made that cascade cheap, which is the same property that made the audit and export paths simple. SnapStart plus the alias made Java on Lambda fast enough to serve reads. And the `source=eventbridge-schedule` tag, set on one line in the schedule target, is what lets the market-hours and holiday guards short-circuit scheduled work while leaving on-demand refreshes untouched. None of these are clever in isolation. The point is that each one was chosen for a reason I could state, and each trade-off was one I was willing to name.
